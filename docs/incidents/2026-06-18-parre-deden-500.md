# Incident — parre-deden.nl site-brede HTTP 500 (18-06-2026)

## Samenvatting
Tussen ~04:00 en ~09:30 (CEST) gaf parre-deden.nl een **site-brede HTTP 500**.
Symptoom dat het meldde: de externe Amelia-notificatie-cron (cronjob.org →
`/wp-admin/admin-ajax.php?action=wpamelia_api&call=/notifications/scheduled/send`)
faalde. In werkelijkheid lag de **hele site** plat. Het probleem herstelde
zichzelf zodra de onderliggende serverstoring voorbij was.

## Hoofdoorzaak
Een **fatale PHP-fout** uit de mailplugin **YaySMTP Pro** (Outlook/Microsoft 365
via OAuth):

```
Uncaught GuzzleHttp\Exception\ConnectException:
cURL error 6: Could not resolve host: login.microsoftonline.com
in wp-content/plugins/yaysmtp-pro/vendor/guzzlehttp/.../CurlFactory.php:200
...
OutlookMsServicesController->__construct()
PluginCore->actionForSmtpsHasAuth()  → hook: init
Time: 2026-06-18 04:57:59
```

### Keten
1. **~04:00 — DNS-/netwerkstoring op de server (XEL-hosting).** De server kon
   `login.microsoftonline.com` niet meer resolven (`cURL error 6`).
2. YaySMTP Pro draait bij **elke request** op de `init`-hook een OAuth-token-
   refresh richting Microsoft.
3. Die call mislukte door de DNS-storing, en YaySMTP **vangt de exception niet
   af** → de `ConnectException` werd een **fatale E_ERROR**.
4. Omdat dit op `init` gebeurt (draait bij élke pagina/AJAX/cron-request), kreeg
   **de hele site** een 500 — niet alleen admin-ajax/Amelia.
5. Toen XEL de DNS/netwerk rond ~07:30 GMT herstelde, slaagde de refresh weer →
   fatal weg → site werkte vanzelf weer.

## Wat het NIET was
- Niet de mcm-security-hardener (Amelia-cron gaf om 03:30 nog 200; deze plugin
  geeft alleen 403's en slaat AJAX/cron/WP-CLI expliciet over).
- Niet Amelia zelf.
- Niet TLS/certificaat (cert geldig t/m 14-08-2026).

## Diagnose-aanwijzingen (voor de toekomst)
- "connection failed" bij cronjob.org + later HTTP 500 → kijk naar **server-/
  netwerkniveau**, niet naar het aangeroepen endpoint.
- Brede, intermitterende 500's op uiteenlopende URL's (echte pagina's,
  robots.txt, favicon) + 429/503 → wijst op een **fatal op een vroege hook
  (init/plugins_loaded)** of resource-uitputting, niet op één kapot endpoint.
- De échte oorzaak staat in het **PHP error log**, niet in de access-log.

## Vervolgacties
- [ ] YaySMTP Pro updaten naar de nieuwste versie (controleer of OAuth-refresh
      inmiddels in try/catch zit).
- [ ] YaySMTP support de stacktrace sturen: token-refresh op `init` moet
      netjes afgevangen worden i.p.v. fataliseren.
- [ ] XEL: oorzaak DNS-resolutiefout rond 04:00 achterhalen + resolver-
      betrouwbaarheid borgen (dit was de trigger).
- [ ] Overweeg uptime-monitoring op de Amelia-cron-URL.
